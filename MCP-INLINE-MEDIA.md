# How Genmax renders images & video inline in Claude chat (MCP)

This explains the **inline media player** — how an image or video generated by the Genmax MCP connector shows up **rendered directly inside the claude.ai conversation**, instead of as a link or a blank box.

It was the hardest thing to get working in the whole project. This doc covers *why* the obvious approaches fail, *what* the working mechanism is, and *how* every moving part fits together, with the exact code.

---

## 1. The problem

When you build a **custom MCP connector** for claude.ai and a tool returns media, the obvious things **do not render inline**:

- ❌ Returning an `image` content block (base64) — claude.ai does **not** inline-render image content from custom connectors.
- ❌ Returning markdown like `![](https://…fal.media/x.png)` — stripped / not rendered.
- ❌ Returning a raw `https://…fal.media/x.mp4` URL as text — shows as a link, not a player.
- ❌ Returning a `video/mp4` blob or `data:` URI in a content block — does not play.

claude.ai only inline-renders media through one official channel: an **MCP Apps UI widget** — a sandboxed iframe your connector ships, which the host (claude.ai) loads and hands the tool result to. The widget renders the media itself. That's the whole trick. (Eromify, the reference product, does exactly this.)

---

## 2. The mechanism, in one paragraph

The connector **registers a UI resource** — a tiny self-contained HTML page at `ui://geoflix/media-v3.html`. Image/video **tools are registered as "app tools"** that carry `_meta.ui.resourceUri` pointing at that widget. When such a tool runs, it returns `structuredContent` containing the media (a base64 image, or a video URL). claude.ai loads the widget in a sandboxed iframe, completes a small **handshake**, then pushes the tool result into the iframe via `postMessage`. The widget's JS finds the media in that payload and injects an `<img>` or `<video>` element — so the media appears **inline, above the assistant's text**.

```
Claude (LLM) calls generate_image / generate_video
        │
        ▼
MCP server tool runs → returns { structuredContent: { kind, url, image } }
        │                        + _meta.ui.resourceUri = ui://geoflix/media-v3.html
        ▼
claude.ai host loads the widget iframe (the registered UI resource)
        │
        ├── handshake:  iframe → ui/initialize → host → initialized
        ▼
host posts the tool result into the iframe (ui/notifications/tool-result)
        │
        ▼
widget JS findMedia() → creates <img>/<video> → renders inline
        │
        └── widget reports its height (ui/notifications/size-changed) so there's no scrollbar
```

## 3. The pieces

### 3a. The widget (the player itself)

A **~3.5 KB hand-written HTML file**: `geoflix-widget/widget.html`. It is deliberately *not* built with the official ext-apps SDK bundle — that bundle is ~376 KB and rendered blank in testing. The hand-written version is tiny and works.

What it does:
1. Shows an animated **"Generating…"** loader until media arrives.
2. Runs a **handshake** with the host (see §4 — this was the bug that broke everything).
3. Listens for `postMessage` events, **scans any incoming payload for media** (`findMedia`), and renders the first hit as `<img>` or `<video>`.
4. **Reports its own height** back to the host so the panel is exactly the media's size — no scrollbar.

Key CSS guarantees no scrollbar even if the host ignores the size report:

```css
#root img,#root video{max-width:100%;max-height:100vh;width:auto;height:auto;display:block;border-radius:12px}
```

`findMedia` is intentionally permissive — it walks the whole message object (up to depth 10) and matches the first thing that looks like media, so it's robust to however the host nests the tool result:

```js
function findMedia(o,d){
  if(o==null||d>10) return null;
  if(typeof o==='string') return /^data:(image|video)\//.test(o) ? {url:o,kind:/^data:video/.test(o)?'video':'image'} : null;
  if(typeof o==='object'){
    if(typeof o.image==='string' && /^data:/.test(o.image)) return {url:o.image, kind:'image'};
    if(typeof o.url==='string' && /^(https?:|data:)/.test(o.url)) return {url:o.url, kind:o.kind||'image'};
    for(var k in o){ var r=findMedia(o[k],d+1); if(r) return r; }
  }
  return null;
}
```

### 3b. The UI resource registration

The widget HTML is served to the host as an MCP **app resource**. The CSP whitelist here is what lets the iframe load media from fal/Google:

```js
registerAppResource(
  server,
  "geoflix-media",
  UI_URI,                                  // "ui://geoflix/media-v3.html"
  { _meta: { ui: { csp: { resourceDomains: [BASE, "https://*.fal.media", "https://*.googleapis.com"] } } } },
  async () => ({ contents: [{ uri: UI_URI, mimeType: RESOURCE_MIME_TYPE, text: WIDGET_HTML }] })
);
```

### 3c. The tools that target the widget

Image/video tools are registered with `registerAppTool` and carry `_meta.ui.resourceUri` — that's what tells claude.ai "render this tool's result inside that widget":

```js
registerAppTool(server, "generate_image", {
  title: "Generate image",
  description: "Generate an image from a text prompt … Renders inline.",
  inputSchema: { prompt: z.string(), model: z.enum([...]).optional() },
  _meta: { ui: { resourceUri: UI_URI } },        // ← links tool → widget
}, async ({ prompt, model }) => { … });
```

The returned object has two parts:
- `structuredContent` — the **machine payload the widget reads** (`{kind, url, image}`).
- `content: [{type:"text", …}]` — a short note for the **LLM**, telling it the media is already shown and to *not* describe it (so Claude doesn't waste a paragraph narrating the image).

## 4. ⭐ The bug that made it all work (the handshake)

This is the single fix that turned a blank box into a working player.

MCP Apps widgets must complete an **initialize handshake** before the host will send the tool result. The widget originally sent the *regular* MCP initialize shape `{capabilities, clientInfo}` — but the **MCP-Apps schema requires a different shape**: `{protocolVersion, appInfo, appCapabilities}`. With the wrong shape, the host never finished init → never sent `ui/notifications/tool-result` → the widget sat on "Generating…" forever.

The correct handshake (from `widget.html`):

```js
function initialize(){
  send({jsonrpc:'2.0',id:++idc,method:'ui/initialize',params:{
    protocolVersion:'2026-01-26',
    appInfo:{name:'geoflix',version:'1.0.0'},
    appCapabilities:{availableDisplayModes:['inline']}
  }});
}
window.addEventListener('message',function(ev){
  var d=ev.data; if(!d||typeof d!=='object') return;
  if(d.method==='ui/notifications/sandbox-proxy-ready') initialize();          // host is ready → init
  if(d.id && d.result && d.result.protocolVersion){                            // host ack'd init
    send({jsonrpc:'2.0',method:'ui/notifications/initialized'});               // → confirm
  }
  var f=findMedia(d,0); if(f) render(f.url,f.kind);                            // any payload may carry media
});
initialize();   // also fire immediately, in case the proxy was ready before we attached the listener
```

> Found by reading `node_modules/@modelcontextprotocol/ext-apps/dist/src/app.js`. If you ever rebuild the widget against a newer ext-apps spec, re-check `protocolVersion` and the param names — a mismatch here silently shows a blank panel.

## 5. Images vs. video — two different delivery paths

The host sandbox is picky about *where* media bytes come from. Images and video are delivered differently on purpose.

### Images → base64 `data:` URI embedded in `structuredContent`
The tool fetches the generated image server-side, base64-encodes it, and ships the bytes **inside** the payload. No cross-origin fetch happens in the iframe at all — most reliable.

```js
const r = await fetch(output);
const mt = r.headers.get("content-type") || "image/jpeg";
imageData = `data:${mt};base64,${Buffer.from(await r.arrayBuffer()).toString("base64")}`;
return {
  structuredContent: { kind: "image", url: proxied(output), image: imageData },  // widget uses `image`
  content: [{ type: "text", text: "Image is displayed in the panel above. Do not describe it — end your turn." }],
};
```

### Video → the **raw `fal.media` URL** (not base64, not the proxy)
Video can't be base64'd into a message (too big), and it needs **HTTP range requests** so `<video>` can stream/seek. fal's CDN supports range requests and the widget CSP whitelists `*.fal.media`, so the widget loads the raw fal URL directly. For non-fal video (e.g. Veo/Google), it falls back to the same-origin proxy.

```js
// in pollVideo():
const url = /fal\.(media|run)/.test(raw) ? raw : proxied(raw);   // fal → raw; others → /api/media proxy
return {
  structuredContent: { url, kind: "video" },
  content: [{ type: "text", text: "Video is displayed in the panel above. Do not describe it — end your turn." }],
};
```

### Why the proxy exists (and why it forwards Range)
claude.ai's sandbox won't load *some* hosts directly. `/api/media` is a same-origin, SSRF-allowlisted proxy. Crucially it **forwards the `Range` header and passes through `206 Partial Content`**, which is what makes proxied `<video>` seekable:

```js
const range = req.headers.get("range");
const r = await fetch(u, { redirect: "follow", headers: range ? { Range: range } : {} });
// … pass through Content-Range / Accept-Ranges and the original status (206) …
return new Response(r.body, { status: r.status, headers });
```

Allowlist (no open proxy): `*.fal.media`, `*.fal.run`, `*.googleapis.com`, `storage.googleapis.com`, `generativelanguage.googleapis.com`.

## 6. Sizing (no scrollbar)

The widget computes its height from the media's **natural aspect ratio** and reports it to the host, mirroring the official SDK. It also re-reports on load, on resize, and on a few timers to catch late layout:

```js
var nw=el&&(el.naturalWidth||el.videoWidth), nh=el&&(el.naturalHeight||el.videoHeight);
if(nw&&nh){ h=Math.ceil(w*nh/nw); }                                   // height from aspect ratio
send({jsonrpc:'2.0',method:'ui/notifications/size-changed',params:{width:w,height:h}});
```

Belt-and-suspenders: the CSS `max-height:100vh` cap (see §3a) means even if the host ignores the size report, the media never produces a scrollbar.

## 7. The build step (don't edit the generated file)

The widget HTML is embedded into the server as a JS string so the serverless function is self-contained. `geoflix-widget/build.js` reads `widget.html` and emits `app/api/[transport]/widget-html.js`:

```js
const html = readFileSync(new URL("./widget.html", import.meta.url), "utf-8");
const out = `// AUTO-GENERATED … do not edit.\nexport const WIDGET_HTML = ${JSON.stringify(html)};\n`;
writeFileSync(new URL("../app/api/[transport]/widget-html.js", import.meta.url), out);
```

**Workflow whenever you change the player:**
```bash
# 1. edit geoflix-widget/widget.html
cd geoflix-widget && node build.js     # regenerates app/api/[transport]/widget-html.js
cd .. && git add -A && git commit -m "…" && git push   # Vercel auto-deploys
```
Never hand-edit `app/api/[transport]/widget-html.js` — it's overwritten by the build.

## 8. ⚠️ The cache gotcha (most common "why isn't my change showing" trap)

**claude.ai caches the widget HTML at connector-connect time.** Any change to the *widget* (its HTML/JS) requires the user to **fully remove and re-add the connector** in claude.ai — toggling it off/on or regenerating the key will **not** refresh the cached widget.

- **Widget change** (HTML/JS, sizing, render logic) → user must **remove + re-add** the connector.
- **Server-only change** (what URL/data a tool returns) → **no** re-add needed; takes effect immediately.
- During debugging, bumping the resource URI version (`media.html` → `media-v2` → `media-v3`) **force-busts** the cache — that's why the current URI is `ui://geoflix/media-v3.html`.

## 9. End-to-end sequence (video example)

```
1. User: "make a 5s clip of a neon city, 9:16"
2. Claude calls tool generate_video({prompt, aspect:"9:16"})
3. Server: POST /api/video/start (fal queue) → returns a job handle
4. Server polls /api/video/status for up to ~45s
      ├─ done in time  → returns structuredContent {url: <raw fal.mp4>, kind:"video"}
      └─ not yet       → returns the handle as text; Claude calls check_video(handle) to finish
5. Host loads widget iframe (ui://geoflix/media-v3.html), CSP allows *.fal.media
6. Handshake: iframe ui/initialize → host ack → iframe ui/notifications/initialized
7. Host posts tool result into iframe
8. widget findMedia() sees {url, kind:"video"} → creates <video controls autoplay loop muted>
9. <video> streams from fal CDN via Range requests → plays inline
10. widget reports size → panel fits the 9:16 clip exactly
11. Claude's text turn ends without describing the video (per the content-block hint)
```

Images are the same minus the polling — the image tool runs synchronously and embeds base64 in step 4.

## 10. File map

| File | Role |
|---|---|
| `geoflix-widget/widget.html` | **The player.** Hand-written iframe: handshake, `findMedia`, render `<img>/<video>`, size reporting. Edit this. |
| `geoflix-widget/build.js` | Inlines `widget.html` → `app/api/[transport]/widget-html.js`. Run after editing the widget. |
| `app/api/[transport]/widget-html.js` | **Auto-generated** `WIDGET_HTML` string. Never edit by hand. |
| `app/api/[transport]/route.js` | MCP server: registers the UI resource + the `generate_image` / `generate_video` / `check_video` app tools; builds `structuredContent`; base64-embeds images; picks raw-fal-vs-proxy for video. |
| `app/api/media/route.js` | Same-origin, allowlisted media proxy with **Range/206 passthrough** (for non-fal video and any host the sandbox won't load directly). |
| `app/api/video/{start,status}/route.js` | The fal video queue the MCP tools call via loopback HTTP. |

## 11. Quick troubleshooting

| Symptom | Likely cause |
|---|---|
| Panel stuck on "Generating…" / blank | Handshake shape wrong (§4), or host never sent the result. Re-check `ui/initialize` params. |
| Your widget edit isn't showing | Cache (§8) — **remove + re-add** the connector; or bump the `media-vN` URI. |
| Image doesn't appear | base64 embed failed (server couldn't fetch the source); check the tool's `fetch(output)`. |
| Video shows but won't play/seek | Range not honored — fal raw URL should be used; for proxied video confirm `/api/media` returns `206` + `Content-Range`. |
| Video host blocked in iframe | Add the host to the CSP `resourceDomains` (§3b) **and** the proxy `ALLOW` list (§5). |
| Forgot to rebuild | You edited `widget.html` but not run `node build.js` — the served `WIDGET_HTML` is stale. |

> Naming note: the product is **Genmax**, but the MCP identifiers in code still use the original `geoflix` slug (`ui://geoflix/…`, server name `geoflix-media`). These are internal IDs — renaming them would force every connected user to re-add the connector (§8), so they're intentionally left as-is.
